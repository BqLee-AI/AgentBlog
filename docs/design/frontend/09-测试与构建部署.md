## 一、测试策略（3 天工期的取舍）

与后端 [13 §一](../13-测试与联调.md) 思路一致：不追覆盖率，追**关键路径有保护**。

| 层级 | 范围 | 工具 | 优先级 |
|------|------|------|--------|
| **纯函数单元** | cn、setServerErrors、token 换算、错误码谓词 | Vitest | 🔴 必做 |
| **hook 测试** | usePostList、useAuth 的状态流转 | Vitest + renderHook | 🟡 时间允许 |
| **组件交互** | 登录表单、API Key 复制弹窗、权限守卫 | Vitest + RTL | 🟡 关键交互 |
| **API 层（mock）** | request 解包、401/402 拦截 | Vitest + MSW | 🔴 必做 |
| **E2E** | 完整登录→发文→对话 | 手工联调 | 🔴 联调阶段 |

> 💡 前端测试投入产出比最高的是**request 层**（契约解包、错误拦截）和**关键交互组件**（API Key 一次性复制、权限守卫）。页面渲染测试性价比低（易碎），靠联调覆盖。

## 二、Vitest 基础

`vite.config.ts` 已配 `test` 字段（见 [01 §2.2](./01-工程化与目录结构.md)）。setup 文件：

```ts
// src/test/setup.ts
import '@testing-library/jest-dom/vitest'
import { afterAll, afterEach, beforeAll } from 'vitest'
import { server } from './server'

// MSW 在测试启动前拦截 fetch

afterEach(() => server.resetHandlers())
afterAll(() => server.close())
```

```bash
bun run test          # 跑全部
bun run test --watch  # 监听
```

## 三、MSW mock fetch（`src/test/server.ts`）

MSW 拦截 `request()` 内部的 fetch，模拟后端响应——包括统一响应结构和错误码。

```ts
// src/test/server.ts
import { setupServer } from 'msw/node'
import { handlers } from './handlers'

export const server = setupServer(...handlers)
```

```ts
// src/test/handlers.ts
import { http, HttpResponse } from 'msw'

export const handlers = [
  // 公开：文章列表
  http.get('/api/posts', ({ request }) => {
    const url = new URL(request.url)
    return HttpResponse.json({
      ok: true,
      data: {
        items: [/* mock post */],
        page: Number(url.searchParams.get('page') ?? 1),
        pageSize: 10,
        total: 1,
      },
    })
  }),

  // 登录
  http.post('/api/auth/login', async ({ request }) => {
    const body = await request.json()
    if (body.username === 'admin') {
      return HttpResponse.json({
        ok: true,
        data: { token: 'test-token', user: { id: 1, username: 'admin', role: 'super_admin', credits: 100000, avatarUrl: null } },
      })
    }
    return HttpResponse.json(
      { ok: false, error: { code: 'UNAUTHORIZED', message: '用户名或密码错误' } },
      { status: 401 },
    )
  }),

  // 401 场景
  http.get('/api/auth/me', ({ request }) => {
    if (!request.headers.get('Authorization')) {
      return HttpResponse.json({ ok: false, error: { code: 'UNAUTHORIZED', message: '未提供认证信息' } }, { status: 401 })
    }
    return HttpResponse.json({ ok: true, data: {/* user */} })
  }),

  // 402 场景（按需 override）
  // http.post('/api/chat', () => HttpResponse.json({ ok:false, error:{ code:'INSUFFICIENT_CREDITS', message:'额度不足' } }, { status: 402 }))
]
```

## 四、request 层测试（必做）

```ts
// src/lib/request.test.ts
import { describe, it, expect, beforeEach } from 'vitest'
import { request } from '@/lib/request'
import { authStore } from '@/lib/auth-store'
import { ApiError } from '@/lib/http-error'

beforeEach(() => authStore.getState().clear())

describe('request', () => {
  it('解包 { ok:true, data }', async () => {
    const data = await request('/api/posts', { public: true })
    expect(data).toHaveProperty('items')
  })

  it('失败抛 ApiError（含 code/message）', async () => {
    await expect(request('/api/posts/no-such')).rejects.toMatchObject({
      name: 'ApiError',
      code: 'NOT_FOUND',
    })
  })

  it('注入 Bearer（有 token 时）', async () => {
    authStore.getState().setAuth('tok', {/* user */})
    await request('/api/auth/me')
    // MSW handler 校验 Authorization 头
  })

  it('401 触发清登录态', async () => {
    authStore.getState().setAuth('bad', {/* user */})
    await expect(request('/api/auth/me')).rejects.toBeInstanceOf(ApiError)
    expect(authStore.getState().token).toBeNull()
  })
})
```

## 六、Vite 构建

### 6.1 构建命令

```bash
bun run build    # tsc -b 类型检查 + vite build
```

产物落在 `apps/web/dist/`，纯静态文件（HTML/JS/CSS/资源）。

### 6.2 构建产物结构

```
dist/
├─ index.html
├─ assets/
│  ├─ index-[hash].js         # 入口 + 公共代码
│  ├─ posts-[hash].js         # 路由懒加载 chunk（阅读端）
│  ├─ admin-[hash].js         # 后台 chunk
│  ├─ chat-[hash].js          # 对话 chunk
│  └─ index-[hash].css
└─ ...
```

> 💡 路由懒加载让首屏（阅读端）chunk 小，后台/对话按需加载。

### 6.3 构建检查清单

- [ ] `tsc -b` 类型检查无错（strict 全开）

- [ ] env 正确（生产 `VITE_API_BASE_URL` 为空，走同源）

## 七、环境变量与同源策略

### 7.1 开发期：Vite proxy（见 [01 §2.2](./01-工程化与目录结构.md)）

前端 `:5173`，通过 proxy 转发 `/api`、`/uploads` 到后端 `:3000`，**零 CORS**。

### 7.2 生产期：Nginx 同源反代

前端静态文件与后端 API 用同一个 Nginx，前端 fetch 写相对路径 `/api/...`，Nginx 转发——**同源，无 CORS**。

```nginx
server {
    listen 443 ssl http2;
    server_name your-blog.com;

    root /opt/agentblog/web/dist;       # 前端静态文件

    # 前端 SPA：所有非文件请求回退到 index.html
    location / {
        try_files $uri $uri/ /index.html;
    }

    # 后端 API + MCP
    location /api/ {
        proxy_pass http://127.0.0.1:3000;
        proxy_http_version 1.1;
        proxy_set_header Host $host;
        proxy_set_header X-Real-IP $remote_addr;
        # SSE 关键：禁缓冲、长超时
        proxy_buffering off;
        proxy_read_timeout 300s;
    }
    location /mcp {
        proxy_pass http://127.0.0.1:3000;
        proxy_buffering off;
        proxy_read_timeout 300s;
    }

    # 上传文件
    location /uploads/ {
        proxy_pass http://127.0.0.1:3000;
        # 或 Nginx 直服静态文件（见后端 14）
    }

    # 静态资源长缓存
    location ~* \.(js|css|png|jpg|svg|woff2)$ {
        expires 1y;
        add_header Cache-Control "public, immutable";
    }
}
```

> 📌 **SSE 流式必读**：`proxy_buffering off` + `proxy_read_timeout 300s` 是 `/api/chat` 流式正常的关键（后端 [14 §五](../14-部署与运维.md) 已强调）。前端 SPA 的 Nginx 配置同样要保留这两项。

### 7.3 前后端不同域的 CORS（备选）

若前后端必须分域名（如前端 CDN），后端开 CORS（后端 [13 §八](../13-测试与联调.md)）：

```ts
// 后端 app.ts
app.use('*', cors({
  origin: env.NODE_ENV === 'production' ? ['https://your-blog.com'] : '*',
  allowHeaders: ['Authorization', 'Content-Type', 'X-API-Key'],
}))
```

前端 `VITE_API_BASE_URL` 填后端地址。**但流式 `useChat` 跨域需后端 CORS 允许流式响应**，配置略繁——优先选同源方案。

## 八、前端部署方式

### 8.1 方式 A：Nginx 托管静态文件（推荐，与后端同机）

```bash
# 构建
cd apps/web
bun run build

# 拷贝到 Nginx 目录
cp -r dist/* /opt/agentblog/web/dist/

# Nginx reload
sudo nginx -s reload
```

### 8.2 方式 B：Docker（与后端 compose 一起）

```dockerfile
# apps/web/Dockerfile
FROM oven/bun:1.1 AS build
WORKDIR /app
COPY package.json bun.lockb ./
RUN bun install --frozen-lockfile
COPY . .
RUN bun run build

FROM nginx:alpine
COPY --from=build /app/dist /usr/share/nginx/html
COPY nginx.conf /etc/nginx/conf.d/default.conf
EXPOSE 80
```

> 💡 若前端独立容器，需配 nginx.conf 做 SPA 回退（`try_files $uri $uri/ /index.html`）+ API 反代到后端容器。

## 九、前后端联调对齐清单（D3）

> 与后端 [13 §八](../13-测试与联调.md) 呼应，前端侧逐项确认。

- [ ] **统一响应结构**：前端 `request()` 能正确解包 `{ ok, data, error }`，所有接口通
- [ ] **错误码表同步**：前端 `ErrorCode` 与后端 [12 §三](../12-统一错误处理与响应规范.md) 完全一致（来自 shared）
- [ ] **Bearer Header**：所有受保护接口带 `Authorization: Bearer`，后端 `authMiddleware` 通过
- [ ] **公开接口**：阅读端 `GET /api/posts`、`/:slug` 不带 token 也能访问
- [ ] **401 联动**：后端返回 401 → 前端自动跳登录；登录后回跳
- [ ] **402 提示**：余额不足 → 前端 toast「额度不足，请联系管理员充值」
- [ ] **字段校验**：前端 zod（shared）与后端 zod-validator 规则一致，错误 details 回填表单
- [ ] **分页结构**：列表统一 `{ items, page, pageSize, total }`（后端 [12 §九](../12-统一错误处理与响应规范.md)）
- [ ] **作者归属**：Agent 文章详情返回 `author.owner`，前端 `AuthorBadge` 展示
- [ ] **slug 永久**：编辑表单无 slug 字段；URL 用 slug 稳定
- [ ] **API Key 明文一次**：签发响应含明文，列表只含 prefix，前端有复制+警示
- [ ] **上传 multipart**：`POST /api/upload` 用 FormData，不手设 Content-Type
- [ ] **流式 `/api/chat`**：`useChat` 能收到 SSE 流，逐字渲染；Nginx 已关 buffering
- [ ] **CORS**（若分域）：后端 allowHeaders 含 `Authorization`/`Content-Type`/`X-API-Key`，origin 含前端域名
- [ ] **类型同步**：`@agentblog/shared` 改动后，前端 `bun run typecheck` 无错
