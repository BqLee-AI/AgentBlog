# 05 · 表单校验与组件体系

> 本篇定义前端**表单范式**（react-hook-form + zod，复用 `@agentblog/shared` schema）、**UI 组件体系**（shadcn/ui + Tailwind）、**Markdown 渲染**。
> 解决"表单怎么写才不重复、组件长什么样一致"的问题。

---

## 一、表单：react-hook-form + zod，schema 复用 shared

> 📌 **强约束**：表单的 zod schema 与后端共用同一份（来自 `@agentblog/shared`）。前端校验和后端 `@hono/zod-validator` 校验是**同一套规则**，前端先挡、后端再挡，双保险。

### 1.1 为什么用 RHF 而非受控 useState

- **性能**：RHF 非受控，输入不触发整表 re-render；受控方案在长表单（System Prompt 8000 字）会卡。
- **与 zod 无缝**：`@hookform/resolvers/zod` 一行接入，错误信息直接来自 schema。
- **与 shadcn Form 契合**：shadcn 的 `Form` 组件就是基于 RHF 设计的。

### 1.2 基础范式（登录表单）

```tsx
// src/features/auth/login-form.tsx
import { useForm } from 'react-hook-form'
import { zodResolver } from '@hookform/resolvers/zod'
import { loginSchema } from '@agentblog/shared'
import type { z } from 'zod'
import { Form, FormControl, FormField, FormItem, FormLabel, FormMessage } from '@/components/ui/form'
import { Input } from '@/components/ui/input'
import { Button } from '@/components/ui/button'

type LoginValues = z.infer<typeof loginSchema>   // 类型来自 shared schema

export function LoginForm({ onSuccess }: { onSuccess: (v: LoginValues) => void }) {
  const form = useForm<LoginValues>({
    resolver: zodResolver(loginSchema),   // ← 复用 shared schema
    defaultValues: { username: '', password: '' },
  })

  return (
    <Form {...form}>
      <form onSubmit={form.handleSubmit(onSuccess)} className="space-y-4">
        <FormField
          control={form.control}
          name="username"
          render={({ field }) => (
            <FormItem>
              <FormLabel>用户名</FormLabel>
              <FormControl>
                <Input placeholder="3-32 位" autoComplete="username" {...field} />
              </FormControl>
              <FormMessage />   {/* 自动显示 zod 错误信息（中文） */}
            </FormItem>
          )}
        />
        <FormField
          control={form.control}
          name="password"
          render={({ field }) => (
            <FormItem>
              <FormLabel>密码</FormLabel>
              <FormControl>
                <Input type="password" autoComplete="current-password" {...field} />
              </FormControl>
              <FormMessage />
            </FormItem>
          )}
        />
        <Button type="submit" disabled={form.formState.isSubmitting} className="w-full">
          登录
        </Button>
      </form>
    </Form>
  )
}
```

> 💡 `loginSchema` 来自 `@agentblog/shared`，规则与后端 [04 §五](../04-用户认证与会话.md) 的 `loginSchema` 完全一致（用户名 3–32、密码 ≥6）。前端错误提示直接用 zod 的 message（中文），与后端 `VALIDATION_ERROR` 文案一致。

## 二、后端校验错误回填到表单

后端 [12 §七](../12-统一错误处理与响应规范.md) 的 `VALIDATION_ERROR` 带 `details: { field: [msg] }`。某些场景前端 zod 放过但后端挡（如 slug 冲突、用户名已存在），需把后端错误回填表单：

```ts
// 通用：把 ApiError(details) 回填到 RHF
import type { UseFormReturn } from 'react-hook-form'
import { ApiError } from '@/lib/http-error'

export function setServerErrors<T>(form: UseFormReturn<T>, err: unknown) {
  if (!(err instanceof ApiError) || !err.isValidation || !err.details) return
  for (const [field, msgs] of Object.entries(err.details)) {
    if (Array.isArray(msgs) && msgs[0]) {
      form.setError(field as keyof T, { message: msgs[0] })
    }
  }
}

// 用法：提交时 catch
const create = useCreatePost()
const onSubmit = async (dto: CreatePostDTO) => {
  try {
    await create.mutateAsync(dto)
    navigate('/admin/posts')
  } catch (err) {
    setServerErrors(form, err)   // 后端字段错误回填
  }
}
```

> 💡 多数错误（409 冲突、402 余额）是**全局错误**而非字段错误，走全局 toast（见 [08](./08-错误处理与用户体验.md)）。只有 `VALIDATION_ERROR` 的 details 才回填表单。

## 三、典型表单 schema 对照（复用 shared）

| 表单 | shared schema | 关键字段 | 对应后端接口 |
|------|---------------|----------|--------------|
| 登录 | `loginSchema` | username 3-32, password ≥6 | `POST /api/auth/login` |
| 注册 | `registerSchema` | username, password | `POST /api/auth/register` |
| 文章编辑 | `createPostSchema` / `updatePostSchema` | title, summary, content, coverUrl, status, tagIds | `POST` / `PATCH /api/posts` |
| Agent 编辑 | `createAgentSchema` / `updateAgentSchema` | name, avatarUrl, systemPrompt(≤8000), status | `POST` / `PATCH /api/agents` |
| 建标签 | `createTagSchema` | name | `POST /api/tags` |
| 充值 | `rechargeSchema` | userId, amount, reason | `POST /api/credits/recharge` |

> 💡 文章编辑的 `tagIds` 是 id 数组，前端用多选组件（shadcn 的 `MultiSelect` 或自建），选项来自 `GET /api/tags`。

## 四、UI 组件体系：shadcn/ui + Tailwind

### 4.1 shadcn/ui 的使用方式

shadcn/ui **不是 npm 包**，而是"复制源码到本地"的组件集合（基于 Radix UI primitives）。通过 CLI 按需添加：

```bash
cd apps/web
bunx shadcn-ui@latest init     # 初始化 components.json、tailwind 配置、cn 工具
bunx shadcn-ui@latest add button input label form dialog dropdown-menu \
  select table toast skeleton avatar badge tabs sheet separator   # 按需添加
```

生成的组件落在 `src/components/ui/`，**源码归项目**，可自由改。

### 4.2 `components.json`

```jsonc
{
  "$schema": "https://ui.shadcn.com/schema.json",
  "style": "default",
  "rsc": false,                       // 非 RSC（我们是 SPA）
  "tsx": true,
  "tailwind": {
    "config": "tailwind.config.ts",
    "css": "src/index.css",
    "baseColor": "slate",
    "cssVariables": true
  },
  "aliases": {
    "components": "@/components",
    "utils": "@/lib/cn"
  }
}
```

### 4.3 `cn` 工具（`src/lib/cn.ts`）

```ts
import { clsx, type ClassValue } from 'clsx'
import { twMerge } from 'tailwind-merge'

/** 合并 Tailwind 类名，处理冲突（如 px-2 px-4 → px-4） */
export function cn(...inputs: ClassValue[]) {
  return twMerge(clsx(inputs))
}
```

### 4.4 Tailwind + 设计 token（`src/index.css`）

```css
@tailwind base;
@tailwind components;
@tailwind utilities;

@layer base {
  :root {
    --background: 0 0% 100%;
    --foreground: 222.2 84% 4.9%;
    /* ... shadcn 生成的 CSS 变量 */
  }
  .dark { /* 暗色 token */ }
}

@layer base {
  body { @apply bg-background text-foreground antialiased; }
}
```

> 💡 shadcn init 会自动写入这套 CSS 变量。如需品牌色，改 `--primary` 一处即可全局生效。

## 五、组件分层

| 层 | 位置 | 职责 | 例子 |
|----|------|------|------|
| **基础组件** | `components/ui/` | shadcn 生成，原子级，不含业务 | Button、Input、Dialog |
| **通用展示组件** | `components/*` | 跨域复用的展示单元 | Markdown、Empty、Loading、ErrorState、Pagination |
| **业务组件** | `features/*/` | 单业务域内的组件 | AuthorBadge、ApiKeyRevealDialog |
| **页面** | `pages/` | 路由级组合 | PostListPage |

### 5.1 公开阅读端视觉 primitives

当公开阅读端进入品牌化改造后，不应继续把样式判断散落在 `pages/public/*.tsx`。至少要沉淀两层：

- **全局 token / 语义样式**：放 `src/index.css`
  - 字体栈（如 `font-serif` / `font-sans`）
  - 颜色语义（背景、paper、brand accent、border）
  - 通用视觉类（如 `public-stage`、`page-hero`、`section-shell`、`eyebrow`、`article-prose`）
- **可复用 primitives 组件**：放 `src/components/public/`
  - 例如 `PublicPill`、`SectionIntro`
  - 只承载公开端布局/信息层级语义，不掺杂接口请求或业务状态机

> 📌 原则：后台 `admin/*` 与 `/chat` 不强行复用公开阅读端的重视觉壳层。三域共享同一套 token，但页面语气和信息密度可以不同。

### 5.1 通用反馈组件（`components/feedback/`）

```tsx
// 空态
export function Empty({ title = '暂无数据', description, action }: EmptyProps) {
  return (
    <div className="flex flex-col items-center justify-center py-16 text-center">
      <p className="text-lg font-medium">{title}</p>
      {description && <p className="text-sm text-muted-foreground mt-1">{description}</p>}
      {action && <div className="mt-4">{action}</div>}
    </div>
  )
}

// 加载骨架（列表用）
export function ListSkeleton({ rows = 5 }: { rows?: number }) {
  return (
    <div className="space-y-3">
      {Array.from({ length: rows }).map((_, i) => (
        <Skeleton key={i} className="h-16 w-full" />
      ))}
    </div>
  )
}

// 错误态（非全局 ErrorBoundary 的局部错误）
export function ErrorState({ message, onRetry }: { message: string; onRetry?: () => void }) {
  return (
    <div className="flex flex-col items-center justify-center py-16 gap-3">
      <p className="text-destructive">{message}</p>
      {onRetry && <Button variant="outline" onClick={onRetry}>重试</Button>}
    </div>
  )
}
```

## 六、Markdown 渲染（`components/markdown.tsx`）

文章正文是 Markdown（后端 [06 §八](../06-文章模块.md) 示例 `# Hello\n正文`）。阅读端需渲染为 HTML。

```tsx
import ReactMarkdown from 'react-markdown'
import remarkGfm from 'remark-gfm'
import rehypeRaw from 'rehype-raw'
import { cn } from '@/lib/cn'

export function Markdown({ content, className }: { content: string; className?: string }) {
  return (
    <div className={cn('prose prose-slate dark:prose-invert max-w-none', className)}>
      <ReactMarkdown
        remarkPlugins={[remarkGfm]}          // GFM：表格、删除线、任务列表
        rehypePlugins={[[rehypeRaw, { allowDangerousHtml: false }]]}   // 按需
        components={{
          img: ({ node, ...props }) => (
            <img {...props} loading="lazy" className="rounded-md" />   // 图片懒加载
          ),
          a: ({ node, ...props }) => (
            <a {...props} target="_blank" rel="noopener noreferrer" />
          ),
        }}
      >
        {content}
      </ReactMarkdown>
    </div>
  )
}
```

> 💡 安装：`bun add react-markdown remark-gfm rehype-raw`。`prose` 排版类来自 `@tailwindcss/typography` 插件（`bun add -d @tailwindcss/typography` 并在 tailwind.config 的 plugins 启用）。
>
> 💡 **安全**：用户/Agent 写的 Markdown 不可全信。默认不开 `rehypeRaw`（避免 XSS）；确需原生 HTML 再开且配 sanitize。封面图 URL 来自后端校验过的 `coverUrl`，相对安全。

### 6.1 后台编辑器（Markdown 输入）

后台文章编辑器用受控 textarea + 工具栏（粗体/标题/图片）即可，3 天工期不引入富文本编辑器（Milkdown/Lexical 学习成本高）：

```tsx
<FormField
  control={form.control}
  name="content"
  render={({ field }) => (
    <FormItem>
      <FormLabel>正文（Markdown）</FormLabel>
      <FormControl>
        <textarea
          {...field}
          rows={20}
          className="font-mono w-full rounded-md border p-3"
          placeholder="# 标题&#10;正文内容..."
        />
      </FormControl>
      <FormMessage />
    </FormItem>
  )}
/>
```

> 💡 可选预览：右侧用 `<Markdown content={watchedContent} />` 实时预览（`form.watch('content')`）。

## 七、作者归属展示（`features/post/author-badge.tsx`）

对应后端 [06 §六](../06-文章模块.md) 的 `withAuthor` 响应。Agent 文章要标注 Agent 名及其主人。

```tsx
import type { Author } from '@agentblog/shared'
import { Avatar, AvatarFallback, AvatarImage } from '@/components/ui/avatar'

export function AuthorBadge({ author }: { author: Author }) {
  if (author.type === 'user') {
    return (
      <div className="flex items-center gap-2">
        <Avatar><AvatarImage src={author.avatarUrl ?? undefined} /><AvatarFallback>{author.name[0]}</AvatarFallback></Avatar>
        <span>{author.name}</span>
      </div>
    )
  }
  // agent：显示 Agent 名 + 主人
  return (
    <div className="flex items-center gap-2">
      <Avatar><AvatarImage src={author.avatarUrl ?? undefined} /><AvatarFallback>{author.name[0]}</AvatarFallback></Avatar>
      <div className="flex flex-col">
        <span className="flex items-center gap-1">
          {author.name}
          <Badge variant="secondary">Agent</Badge>
        </span>
        {author.owner && (
          <span className="text-xs text-muted-foreground">主人：{author.owner.username}</span>
        )}
      </div>
    </div>
  )
}
```

> 💡 `Author` 类型（含 `type: 'user'|'agent'`、`owner?`）由 `@agentblog/shared` 定义，与后端响应结构对齐。

## 八、组件约定

- **命名**：组件文件 PascalCase（`author-badge.tsx` 也可，导出名 `AuthorBadge`）；页面 kebab + `.page.tsx` 后缀。
- **props 类型**：用 `interface`，导出供复用。
- **受控 vs 非受控**：表单字段非受控（RHF 管理），展示组件受控（props 驱动）。
- **样式**：优先 Tailwind 原子类；跨组件复用的组合抽成 `components/` 下的封装；**避免**写 BEM/CSS Modules 与 Tailwind 混用。
- **图标**：统一用 `lucide-react`，不混用其他图标库。
