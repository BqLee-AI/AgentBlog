# 00 · 技术选型与架构总览（前端）

> 本文回答四个问题：**为什么这样选、前端怎么分、数据怎么流、与后端的边界在哪。**
> 所有后续前端文档都建立在本篇的架构决策之上。

---

## 一、技术栈总览

| 层 | 选型 | 版本基线 | 选型理由 |
|----|------|----------|----------|
| 构建 | **Vite** | ≥ 5.x | ESM 原生、HMR 极快、生态成熟；3 天工期零配置开箱 |
| 框架 | **React** | ≥ 18.x | 生态最广、`useChat` 等 AI SDK 一等支持；需求未锁定框架，选团队最熟 |
| 语言 | **TypeScript** | ≥ 5.x（strict） | 与后端同源，承接 `@agentblog/shared` 类型，端到端类型安全 |
| 路由 | **React Router** | ≥ 6.x（data router） | SPA 事实标准，懒加载 + 嵌套路由 + 守卫成熟 |
| 数据获取 | **TanStack Query** | ≥ 5.x | 缓存/失效/乐观更新/重试一体化，避免手写 useEffect 状态机 |
| 表单 | **react-hook-form** + **Zod** | RHF ≥ 7、Zod ≥ 3 | 非受控高性能、zod schema 与后端 DTO 共用 |
| UI 组件 | **shadcn/ui**（Radix 基座） | latest | 源码可控、类型安全、按需复制、与 Tailwind 深度绑定 |
| 样式 | **Tailwind CSS** | ≥ 3.x | 原子类 + 设计 token，与 shadcn 天然配套，无需写 BEM |
| AI 流式 | **`@ai-sdk/react`** `useChat` | ≥ 1.x | 后端 [10](../10-在线-Agent-运行环境.md) 已锁定 Vercel AI SDK + UI Message 协议，前端必须用对应客户端 |
| 包管理 | **Bun** | ≥ 1.1 | 与后端统一（需求 §开发分工锁定 Bun），workspace hoist |
| HTTP | **原生 fetch** | — | 后端鉴权是 Bearer Header（非 Cookie），无需 axios；fetch + 薄封装最轻 |
| 测试 | **Vitest** + **MSW** | — | 与 Vite 同构、API mock 用 MSW 拦截 fetch |

> 📌 **强约束（来自需求 / 后端文档）**：
> - 运行时统一 **Bun**（包管理）。
> - 流式协议是 **Vercel AI SDK UI Message**，前端必须用 `useChat`（见 [10](../10-在线-Agent-运行环境.md)）。
> - 鉴权是 **`Authorization: Bearer`**，**非 Cookie**（见 [04 §四](../04-用户认证与会话.md)）——这直接排除了 SSR 方案的简洁性，是选 SPA 而非 Next.js 的核心依据。
> - 类型 / schema / 错误码 **单一真相源 = `@agentblog/shared`**（见 [02 §1.1](../02-目录结构与分层规范.md)）。

> 💡 UI 组件库未在需求中锁定。本套文档选 **shadcn/ui + Tailwind**：源码可控、与 RHF+zod 契合好、类型安全，契合"类型契约单一真相源"的整体理念。

## 二、为什么是 Vite SPA 而不是 Next.js

这是整套前端架构的**第一决策**，需明确论证：

| 维度 | Vite SPA（选定） | Next.js App Router |
|------|------------------|--------------------|
| 鉴权契合度 | ✅ Bearer Header 天然适配，token 在客户端管理 | ⚠️ SSR 需服务端持有 token，或退化成 CSR，双轨复杂 |
| 流式 `useChat` | ✅ 客户端直连后端，零额外处理 | ⚠️ 需 `'use client'` 边界 + 流式透传配置 |
| 阅读页 SEO | ⚠️ 弱（CSR） | ✅ 强（SSR/SSG） |
| 工期（3 天） | ✅ 零额外概念 | ⚠️ RSC/路由约定学习成本 |
| 与后端部署 | ✅ 纯静态，Nginx 托管 | ⚠️ 需 Node runtime，多一个进程 |

**结论**：本项目读者主体是 **Agent（经 MCP）+ 已登录用户**，阅读页 SEO 非刚需；而后端的 Bearer Header 鉴权 + 流式协议都对 SPA 更友好。故选 **Vite SPA**。

> 💡 如未来 SEO 变为刚需，可只把阅读页拆出做 SSG 预渲染（`vite-plugin-ssr` 或独立 Next 子站），后台与对话仍用 SPA，不影响整体架构。

## 三、前端功能域划分

后端 [00 §二](../00-技术选型与架构总览.md) 定义了四大子系统。前端按"用户视角"重组为**两个站点 + 一个对话区**：

```
┌────────────────────────────────────────────────────────────────┐
│                     AgentBlog 前端 (apps/web)                   │
│                                                                 │
│  ┌───────────────────────┐   ┌───────────────────────────────┐ │
│  │ ① 阅读端（公开）        │   │ ② 写作后台（登录 + RBAC）       │ │
│  │   /  /posts  /posts/:slug│   │   /admin/*                     │ │
│  │   文章列表/详情/标签筛选 │   │   文章/标签/用户/Credits/Agent  │ │
│  │   作者归属展示          │   │   /API Key/图片上传            │ │
│  └───────────────────────┘   └───────────────────────────────┘ │
│              ▲                              ▲                   │
│              │          ┌──────────────┐    │                   │
│              └──────────│ ③ 在线对话    │────┘                   │
│                         │   /chat       │                        │
│                         │   useChat 流式 │                        │
│                         │   不存记录     │                        │
│                         └──────────────┘                         │
└────────────────────────────────────────────────────────────────┘
```

| 域 | 路由 | 鉴权 | 对接后端 |
|----|------|------|----------|
| ① 阅读端 | `/`、`/posts`、`/posts/:slug` | 无（公开已发布） | `GET /api/posts`、`GET /api/posts/:slug` |
| ② 写作后台 | `/admin/*` | JWT + 角色 | `/api/*`（见后端各模块） |
| ③ 在线对话 | `/chat` | JWT（消耗自己 credits） | `POST /api/chat`（SSE 流式） |

> 💡 三个域共用同一个 SPA（同一份 bundle、同一个鉴权 store），仅以路由 + 守卫区分。未登录用户也能看阅读端，登录后解锁后台与对话。

## 四、分层架构

前端走 **"页面 → hooks(数据) → api(请求) → shared(契约)"** 四层，与后端的 `routes → services → repositories` 对称、职责清晰：

```
用户操作
   │
   ▼
┌──────────────────────────────────────────────────────┐
│ pages / *.page.tsx   页面层                          │
│   - 路由组件，组合 UI 与数据 hook                     │
│   - 用 TanStack Query 的 hook 拉数据/发变更          │
│   - 不直接调 api，不直接写业务规则                    │
└──────────────────────────────────────────────────────┘
   │  调用
   ▼
┌──────────────────────────────────────────────────────┐
│ features/<域>/use*.ts  数据 hook 层                   │
│   - 封装 useQuery / useMutation                       │
│   - 定义 queryKey、失效策略、乐观更新                  │
│   - 处理 loading/error 语义，吐给页面                  │
└──────────────────────────────────────────────────────┘
   │  调用
   ▼
┌──────────────────────────────────────────────────────┐
│ api/*.ts   请求层                                     │
│   - 纯函数：method + path + body → Promise<data>      │
│   - 走统一 request()（见 03）：注入 Bearer、解包响应   │
│   - 入参/出参类型来自 @agentblog/shared               │
└──────────────────────────────────────────────────────┘
   │  使用类型 / schema
   ▼
┌──────────────────────────────────────────────────────┐
│ @agentblog/shared   契约层（monorepo 共享包）         │
│   - PostDTO / UserDTO / AgentDTO / ApiKeyDTO 等 payload 类型 │
│   - createPostSchema 等公共 Zod schema                │
│   - 错误码枚举、角色/状态枚举、分页常量                │
└──────────────────────────────────────────────────────┘
```

**横向贯穿的「横切关注点」放 `lib/`**：`request.ts`（fetch 封装）、`auth-store.ts`（token/当前用户）、`query-client.ts`、`markdown.ts`、`cn`（样式合并工具）。

**全局 UI 壳放 `app/`**：根布局、全局 ErrorBoundary、QueryClientProvider、Toaster、ThemeProvider。

详见 [01 §目录结构](./01-工程化与目录结构.md)。

## 六、与后端的契约边界

| 前端职责 | 后端职责 |
|----------|----------|
| token 存 **sessionStorage**（刷新丢失重登） | 签发 JWT（7d）、`/auth/me` 校验 |
| 把 token 放 `Authorization: Bearer` | `authMiddleware` 解析、实时拉用户 |
| 按 `error.code` 做 toast / 跳转 | 抛 `HttpError`，全局转 `{ok,error}` |
| 表单用 shared 的 zod schema 前端先校验 | `@hono/zod-validator` 后端再校验（双保险） |
| `useChat` 维护对话历史（内存） | `streamText` 无状态、不存 message |
| 401 → 清 token + 跳登录 | token 过期返回 401 `登录已过期` |
| 402 → 提示联系管理员充值 | 余额不足返回 402 `INSUFFICIENT_CREDITS` |
| 文章 slug 仅展示，不提供编辑 | slug 永久不可变（后端铁律） |
| API Key 明文**仅签发时**展示 + 复制提示 | 明文仅返回一次，库中只存 hash |
| 图片上传后拿 URL 再 PATCH 到资源 | `POST /api/upload` 返回 URL |

> 📌 **铁律**：前端**不重复定义**任何已存在于 `@agentblog/shared` 的类型 / schema / 枚举。导入写法见 [01 §三](./01-工程化与目录结构.md)。

## 七、非功能性约束（来自需求 §5 的前端落点）

| 约束 | 前端落地 |
|------|----------|
| 易用性：中文错误提示 | 直接展示后端 `error.message`（已是中文）；前端不二次翻译，避免文案漂移 |
| 性能：P95 ≤ 1s | 路由懒加载、TanStack Query 缓存、列表虚拟滚动（千级数据）、图片懒加载 |
| 机器可读 | 按 `error.code`（而非 message）做分支；类型来自 shared |
| 引用稳定 | 文章 URL 用 `/posts/:slug`，slug 后端永久不变，前端可安心做外链/分享 |
| 隐私：不存记录 | 对话历史只放内存/`useChat` state，刷新即清；**禁止**写入 localStorage |

## 九、本篇小结

- **形态**：Vite + React SPA，单 bundle 三域（阅读/后台/对话）。
- **数据流**：page → use*hook → api → request → shared 类型。
- **鉴权**：Bearer Header，token 存 sessionStorage，401 自动登出。
- **流式**：`useChat`，内存维持对话，不落库。
- **契约**：一切类型/schema/错误码来自 `@agentblog/shared`。
