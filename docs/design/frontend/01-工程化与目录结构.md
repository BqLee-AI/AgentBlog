## 二、TypeScript 配置（继承根 base）

monorepo 已有根 `tsconfig.base.json`（见后端 [01 §5.1](../01-项目初始化与工程化.md)）。前端 `extends` 继承，保持 strict / Bundler / verbatimModuleSyntax 全局一致。

### 2.1 `apps/web/tsconfig.json`

```jsonc
{
  "extends": "../../tsconfig.base.json",
  "compilerOptions": {
    "lib": ["ES2022", "DOM", "DOM.Iterable"],   // 前端需要 DOM
    "jsx": "react-jsx",
    "types": ["vite/client", "vitest/globals"],

    "baseUrl": ".",
    "paths": {
      "@/*": ["src/*"],
      "@agentblog/shared": ["../../packages/shared/src/index.ts"],
      "@agentblog/shared/*": ["../../packages/shared/src/*"]
    },

    "useDefineForClassFields": true,
    "noEmit": true                              // Vite 负责构建产物
  },
  "include": ["src/**/*.ts", "src/**/*.tsx"],
  "exclude": ["node_modules", "dist"],
  "references": [
    // 若用 tsc -b 做项目引用，指向 shared
    { "path": "../../packages/shared" }
  ]
}
```

**为什么这样配**：
- `extends` 根基线：strict / Bundler / verbatimModuleSyntax 与后端完全一致，单一真相源。
- `lib` 加 `DOM`：前端要用 `document`、`fetch`、`File` 等。
- `@/*` + `@agentblog/shared` 别名：避免深层相对路径；shared 包指向源码，编辑器跳转直达，无需先 build。

### 2.2 Vite 别名同步（`vite.config.ts`）

> ⚠️ **Vite 的路径解析与 tsconfig 的 `paths` 是两套独立机制**，必须同时配置，否则运行时 import 会报模块找不到。

```ts
// apps/web/vite.config.ts
import { defineConfig } from 'vite'
import react from '@vitejs/plugin-react'
import { fileURLToPath, URL } from 'node:url'

export default defineConfig({
  plugins: [react()],
  resolve: {
    alias: {
      '@': fileURLToPath(new URL('./src', import.meta.url)),
      // shared 包走 workspace 解析即可，无需额外别名；若要直链源码：
      // '@agentblog/shared': fileURLToPath(new URL('../../packages/shared/src/index.ts', import.meta.url)),
    },
  },
  server: {
    port: 5173,
    // 把 /api、/mcp、/uploads 代理到后端，避免开发期 CORS
    proxy: {
      '/api': { target: 'http://localhost:3000', changeOrigin: true },
      '/uploads': { target: 'http://localhost:3000', changeOrigin: true },
    },
  },
  test: {
    globals: true,
    environment: 'jsdom',
    setupFiles: ['./src/test/setup.ts'],
  },
})
```

> 💡 开发期用 Vite proxy 把 `/api` 转发到后端 `:3000`，前端代码里 fetch 写**相对路径** `/api/...`，零 CORS 配置。生产期则靠 Nginx 同源反代（见 [09](./09-测试与构建部署.md)）。

## 三、引入 `@agentblog/shared`（契约单一真相源）

> 📌 这是端到端类型安全的关键。前端**禁止**重定义已存在于 shared 的类型 / schema / 枚举。

```ts
// 前端任意位置
import type { Post, User, ApiKey } from '@agentblog/shared'        // payload 类型
import { createPostSchema, ErrorCode, Role, PostStatus } from '@agentblog/shared'  // schema、枚举、错误码
```

shared 包当前导出（见后端 [02 §3.2](../02-目录结构与分层规范.md)）：

| 导出 | 用途 | 前端使用位置 |
|------|------|--------------|
| `types.ts`：`Post` / `User` / `Agent` / `ApiKey` / `CreditLog` / `Paginated<T>` | API 响应 payload 类型 | api 层返回类型、组件 props |
| `schemas.ts`：`createPostSchema` / `loginSchema` / `createAgentSchema` 等 | 前后端共用校验 | RHF 的 `zodResolver` |
| `codes.ts`：`ErrorCode` 枚举 | 机器可读错误码 | request 层错误分支（见 [03](./03-API-层与请求封装.md)） |
| `constants.ts`：`Role` / `PostStatus` / `PAGE_SIZE` | 角色权限 / 状态 / 分页 | 守卫、列表、按钮显隐 |

> 💡 若某个 schema 后端还没上提到 shared，前端**先用本地 schema 占位并加 TODO**，待后端补齐后替换为 shared 导入——不要长期双份并存。

## 五、分层职责矩阵

| 层 | 位置 | 允许做 | 禁止做 |
|----|------|--------|--------|
| **契约** | `@agentblog/shared` | 定义 payload 类型、共用 zod schema、错误码、枚举 | 业务逻辑、React 组件、fetch 调用 |
| **请求层** | `api/*.api.ts` / `lib/request.ts` | 纯函数：method+path+body → `Promise<data>`；注入 Bearer；解包响应 | 写 React、维护业务 state、调 hook |
| **数据 hook 层** | `features/*/use-*.ts` | 封装 useQuery/useMutation；定义 queryKey、失效、乐观更新 | 直接调 fetch；写 UI |
| **页面层** | `pages/*.page.tsx` | 组合 hook + 组件；路由参数处理；触发导航 | 直接调 api；维护可放 hook 的 state |
| **组件层** | `components/*`、`features/*` | 纯展示 / 受控交互；props 驱动 | 直接发请求（除非是自包含组件如上传） |
| **横切库** | `lib/*` | 纯函数工具（request/auth-store/cn/markdown） | 含业务流程 |

> 📌 **依赖铁律**：`pages → features(use*) → api → lib(request) → shared`，不得反向。`features/post` 可以用 `features/tag` 的 hook（跨域允许，注意循环依赖）。`pages` 之间不互相 import 实现，只通过路由跳转。

## 六、环境变量（Vite 约定）

> 📌 Vite 只对 `VITE_` 前缀的变量注入客户端。前端 env **不含任何后端密钥**（JWT_SECRET / OPENAI_API_KEY 等绝不能出现在前端）。

### 6.1 `.env.development`（提交）

```bash
# 后端 API 基址。开发期留空（走 Vite proxy 同源）
VITE_API_BASE_URL=
```

### 6.2 `.env.production`（提交，不含敏感）

```bash
# 生产环境前端通过同源 Nginx 反代访问后端，留空即可
# 若前后端不同域，填后端地址，并确保后端 CORS 已放行
VITE_API_BASE_URL=
```

### 6.3 类型化读取（`src/config/env.ts`）

```ts
import { z } from 'zod'

const envSchema = z.object({
  VITE_API_BASE_URL: z.string().default(''),
})

// 构建期解析，失败立即抛（Vite 构建阶段）
const parsed = envSchema.parse(import.meta.env)

export const env = {
  apiBaseUrl: parsed.VITE_API_BASE_URL,
}
```

> 💡 用 zod 校验 env，与后端 [01 §7.3](../01-项目初始化与工程化.md) 思路一致——启动即失败胜过运行时 undefined。

### 6.4 `vite-env.d.ts`（类型补充）

```ts
/// <reference types="vite/client" />

interface ImportMetaEnv {
  readonly VITE_API_BASE_URL: string
}
interface ImportMeta {
  readonly env: ImportMetaEnv
}
```
