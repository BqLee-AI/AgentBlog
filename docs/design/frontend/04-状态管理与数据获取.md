# 04 · 状态管理与数据获取

> 本篇定义前端的**状态分类**、**服务端状态**（TanStack Query）与**客户端状态**（auth store）的边界与用法，以及**缓存失效策略**。
> 解决"什么数据该放哪、什么时候刷新"的问题。

---

## 一、状态分类

前端状态分三类，各用各的工具，**不要混用**：

| 类型 | 例子 | 工具 | 特征 |
|------|------|------|------|
| **服务端状态** | 文章列表、用户列表、Agent、Credits 流水 | TanStack Query | 来自后端、可失效、多组件共享、自动缓存 |
| **客户端全局状态** | token、当前用户、登录态 | auth store（zustand 或 context） | 跨路由持久、与鉴权强耦合 |
| **客户端局部状态** | 表单输入、弹窗开关、筛选条件 | useState / RHF | 单组件或单表单内 |

> 📌 **铁律**：服务端数据**只走 TanStack Query**，不要用 `useState + useEffect + fetch` 手写状态机——那是 bug 重灾区（竞态、重复请求、无缓存、无失效）。

---

## 二、为什么不全用 Redux / Zustand 管所有状态

后端数据本质是**服务端状态的缓存**，它有独特需求：缓存、失效、重试、乐观更新、请求去重。Redux/Zustand 是通用客户端状态容器，没有这些能力，要手写一遍等于重新发明 TanStack Query。

| 需求 | TanStack Query | Redux/Zustand |
|------|----------------|---------------|
| 缓存 + 去重 | 内置 | 手写 |
| 失效（mutation 后刷新） | `invalidateQueries` | 手写 |
| 乐观更新 | `onMutate` | 手写 |
| 请求竞态 | `signal` 自动取消 | 手写 |
| 加载/错误态 | `isLoading/isError` 内置 | 手写 reducer |

**结论**：服务端状态交 Query，客户端全局状态（只有 auth）用 zustand，其余 useState/RHF。

---

## 三、auth store（`src/lib/auth-store.ts`）

唯一需要跨路由持久的客户端状态：token + 当前用户 + 登录态。用 zustand（轻量、无 Provider、TS 友好）。

```ts
import { create } from 'zustand'
import type { UserDTO } from '@agentblog/shared'

const TOKEN_KEY = 'agentblog_token'
const TOKEN_STORAGE = sessionStorage   // 📌 后端 [04 §四] 建议内存或 sessionStorage，不用 localStorage

interface AuthState {
  token: string | null
  user: UserDTO | null
  status: 'idle' | 'loading' | 'authenticated' | 'unauthenticated' | 'error'

  setAuth: (token: string, user: UserDTO) => void
  setUser: (user: UserDTO) => void
  clear: () => void
}

export const useAuthStore = create<AuthState>((set) => ({
  token: TOKEN_STORAGE.getItem(TOKEN_KEY),
  user: null,
  status: 'idle',

  setAuth: (token, user) => {
    TOKEN_STORAGE.setItem(TOKEN_KEY, token)
    set({ token, user, status: 'authenticated' })
  },
  setUser: (user) => set({ user }),
  clear: () => {
    TOKEN_STORAGE.removeItem(TOKEN_KEY)
    set({ token: null, user: null, status: 'unauthenticated' })
  },
}))

// 非 hook 场景（request.ts、require-auth 等）取值用 getState
export const authStore = useAuthStore
```

### 3.1 为什么用 sessionStorage 而非 localStorage

后端 [04 §四](../04-用户认证与会话.md) 明确建议：**避免 localStorage**（持久化增大 XSS 暴露面）。token 存 `sessionStorage`：
- 关闭标签页即清，降低泄露窗口。
- 刷新页面不丢（同一 tab 内有效）。
- 多 tab 独立登录态，避免串号。

> 💡 如需"刷新不丢 + 关浏览器才清"的折中，可加内存镜像：sessionStorage 读一次进内存，之后只用内存值。v1 直接 sessionStorage 已足够。

## 四、AuthProvider：启动时校验 token

应用启动若有 token，调 `/api/auth/me` 校验有效性并取当前用户。`status` 流转：`idle → loading → authenticated / unauthenticated / error`，供 `RequireAuth` 守卫消费（见 [02 §四](./02-路由设计与权限守卫.md)）。

- `unauthenticated`：无 token，或 401 已被 `request()` 清态。
- `error`：`/auth/me` 非 401 失败（网络/5xx 等），保留 token，交给守卫层显示可重试错误态。

```tsx
// src/features/auth/use-auth.ts
import { useEffect } from 'react'
import { useAuthStore, authStore } from '@/lib/auth-store'
import { authApi } from '@/api/auth.api'
import { queryKeys } from '@/lib/query-keys'
import { queryClient } from '@/lib/query-client'
import { ApiError } from '@/lib/http-error'

export function useAuth() {
  return useAuthStore()
}

// 启动校验：挂在 RootProviders 内
export function AuthProvider({ children }: { children: React.ReactNode }) {
  const token = useAuthStore((s) => s.token)
  const status = useAuthStore((s) => s.status)

  useEffect(() => {
    if (!token) {
      if (status !== 'unauthenticated') {
        authStore.setState({ status: 'unauthenticated' })
      }
      return
    }

    // 仅在 idle 时启动校验；authenticated / loading / error 都不重复打 /auth/me
    if (status !== 'idle') return
    authStore.setState({ status: 'loading' })

    let cancelled = false

    authApi.me()
      .then((user) => {
        if (cancelled) return
        authStore.getState().setAuth(token, user)
        queryClient.setQueryData(queryKeys.me, user)
      })
      .catch((err: unknown) => {
        if (cancelled) return

        // 401：request() 已处理（clear token + 清缓存 + 跳登录），这里无需重复 clear
        if (err instanceof ApiError && err.isUnauthorized) return

        // 非 401：保留 token，进入可重试错误态，避免 /admin -> /login -> /admin 循环
        authStore.setState({ status: 'error' })
      })

    return () => {
      cancelled = true
    }
  }, [token, status])

  return <>{children}</>
}
```

> 💡 `/auth/me` 既校验 token 又取最新 user（含 credits）。把结果顺手写入 `queryKeys.me` 缓存，后台顶栏显示当前用户可直接读缓存。只有 401 才代表认证失效；网络错误/5xx 不应伪装成“未登录”。

## 五、QueryClient 配置（`src/lib/query-client.ts`）

```ts
import { QueryClient } from '@tanstack/react-query'
import { ApiError } from '@/lib/http-error'

export const queryClient = new QueryClient({
  defaultOptions: {
    queries: {
      staleTime: 30_000,          // 30s 内不重复请求（千级数据变化不频繁）
      gcTime: 5 * 60_000,         // 5min 后回收
      retry: (failureCount, error) => {
        // 401/403/404/402/400 不重试（重试也没用）
        if (error instanceof ApiError && [400, 401, 402, 403, 404, 409].includes(error.status)) {
          return false
        }
        return failureCount < 2   // 其余（500/网络）重试 2 次
      },
      refetchOnWindowFocus: false, // 博客场景无需聚焦自动刷新
    },
    mutations: {
      retry: false,                // 写操作不自动重试（避免重复创建）
    },
  },
})
```

> 💡 **mutation 不自动重试**：创建文章重试会导致重复创建。幂等的可显式开 retry。

## 六、集中式 queryKey 工厂（`src/lib/query-keys.ts`）

> 📌 **queryKey 集中管理**，避免散落各处的字符串数组拼错。一处定义，处处复用，失效时也只引这一个工厂。

```ts
export const queryKeys = {
  // auth
  me: ['me'] as const,

  // posts
  posts: {
    all: ['posts'] as const,
    list: (params: Record<string, unknown>) => ['posts', 'list', params] as const,
    detail: (slug: string) => ['posts', 'detail', slug] as const,
    detailById: (id: number) => ['posts', 'by-id', id] as const,
  },

  // tags
  tags: {
    all: ['tags'] as const,
  },

  // users
  users: {
    all: ['users'] as const,
    list: (params: Record<string, unknown>) => ['users', 'list', params] as const,
  },

  // agent（每用户≤1）
  agent: {
    mine: ['agent', 'mine'] as const,
  },

  // api keys
  apiKeys: {
    byAgent: (agentId: number) => ['api-keys', 'agent', agentId] as const,
  },

  // credits
  credits: {
    myLogs: (params: Record<string, unknown>) => ['credits', 'my-logs', params] as const,
    logs: (userId: number, params: Record<string, unknown>) => ['credits', 'logs', userId, params] as const,
  },
} as const
```

## 七、数据 hook 范式（`src/features/*/use-*.ts`）

每个业务域的 hook 封装 useQuery/useMutation，页面层只调 hook。

### 7.1 查询 hook（文章列表）

```ts
// src/features/post/use-posts.ts
import { useQuery } from '@tanstack/react-query'
import { postsApi } from '@/api/posts.api'
import { queryKeys } from '@/lib/query-keys'
import type { ListPostsQuery } from '@agentblog/shared'

export function usePostList(params: ListPostsQuery) {
  return useQuery({
    queryKey: queryKeys.posts.list(params),
    queryFn: ({ signal }) => postsApi.listPublic(params, signal),
    placeholderData: (prev) => prev,   // 翻页时保留旧数据，避免闪烁
  })
}

export function usePost(slug: string) {
  return useQuery({
    queryKey: queryKeys.posts.detail(slug),
    queryFn: ({ signal }) => postsApi.getBySlug(slug, signal),
    enabled: !!slug,
  })
}
```

### 7.2 变更 hook（创建文章）+ 缓存失效

```ts
import { useMutation, useQueryClient } from '@tanstack/react-query'
import { postsApi } from '@/api/posts.api'
import { queryKeys } from '@/lib/query-keys'
import type { CreatePostDTO } from '@agentblog/shared'
import { toast } from 'sonner'   // 或 shadcn 的 useToast

export function useCreatePost() {
  const qc = useQueryClient()
  return useMutation({
    mutationFn: (dto: CreatePostDTO) => postsApi.create(dto),
    onSuccess: (post) => {
      // 失效列表（让列表重新拉取含新文章）
      qc.invalidateQueries({ queryKey: queryKeys.posts.all })
      toast.success('文章创建成功')
    },
  })
}
```

### 7.3 乐观更新（删除文章）

```ts
export function useDeletePost() {
  const qc = useQueryClient()
  return useMutation({
    mutationFn: (id: number) => postsApi.remove(id),
    onMutate: async (id: number) => {
      // 取消进行中的列表查询，避免覆盖乐观结果
      await qc.cancelQueries({ queryKey: queryKeys.posts.all })
      // 快照当前列表
      const prev = qc.getQueriesData<{ items: PostDTO[] }>({ queryKey: queryKeys.posts.all })
      // 乐观移除
      qc.setQueriesData<{ items: PostDTO[] }>({ queryKey: queryKeys.posts.all }, (old) =>
        old ? { ...old, items: old.items.filter((p) => p.id !== id) } : old,
      )
      return { prev }
    },
    onError: (_err, _id, ctx) => {
      // 失败回滚
      ctx?.prev.forEach(([key, data]) => qc.setQueryData(key, data))
      toast.error('删除失败')
    },
    onSettled: () => {
      qc.invalidateQueries({ queryKey: queryKeys.posts.all })
    },
  })
}
```

## 八、缓存失效策略总表

| 操作 | 失效的 queryKey | 原因 |
|------|-----------------|------|
| 创建文章 | `posts.all` | 列表需含新文章 |
| 更新文章 | `posts.detail(slug)`、`posts.all`、`posts.by-id(id)` | 详情与列表都要更新 |
| 删除文章 | `posts.all` | 列表移除该项 |
| 登录 | `me`（直接 setQueryData） | 写入当前用户 |
| 登出 | `queryClient.clear()` | 清所有用户相关缓存，防残留 |
| 签发 API Key | `apiKeys.byAgent(agentId)` | 列表加新 Key |
| 吊销 API Key | `apiKeys.byAgent(agentId)` | 列表状态变更 |
| 更新 Agent | `agent.mine` | 头像/prompt 变更 |
| 充值 Credits | `users.list`（admin 视角）、`me`（自己余额） | 余额变更 |
| 对话扣费后 | `me`（重拉取最新 credits） | 余额可能变化（见 [07](./07-在线对话与流式实现.md)） |

> 💡 **失效粒度**：能精确到 detail 就别 `all`；但 mutation 后列表通常要 invalidate（顺序/总数变了）。`all` 是兜底，`detail` 是优化。

## 九、与 React 18 的并发渲染

TanStack Query v5 与 React 18 的 Suspense 可配合，但本项目**默认不开 Suspense 模式**——用 `isLoading/isPending` + Skeleton 更可控（部分骨架 vs 全屏 fallback）。如需开：

```ts
useQuery({ ..., suspense: true })  // 可选
```

> 💡 千级数据 + 3 天工期，Skeleton + `placeholderData` 已足够，不引入 Suspense 的额外心智。
