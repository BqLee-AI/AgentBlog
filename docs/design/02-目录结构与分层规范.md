## 一、Monorepo 顶层结构

monorepo 采用 Bun Workspaces，三个工作区 + 一个根基线配置。完整顶层树如下：

```
AgentBlog/                          # monorepo 根
├─ package.json                     # workspaces 配置 + 跨包脚本（dev/build/typecheck/lint）
├─ tsconfig.base.json               # 全局 TS 基线（strict / Bundler / verbatimModuleSyntax）
├─ bunfig.toml                      # Bun 配置（可选）
├─ .gitignore
├─ .editorconfig
├─ README.md
│
├─ docs/                            # 本套设计文档（前后端共阅）
│  ├─ README.md
│  └─ 00~14 各篇设计文档
│
├─ packages/
│  └─ shared/                       # @agentblog/shared：前后端共享契约
│     ├─ package.json               # exports 指向 ./src/index.ts（不编译，直接消费 TS 源）
│     ├─ tsconfig.json              # extends 根 base
│     └─ src/
│        ├─ index.ts                # 统一出口（re-export 下述模块）
│        ├─ types.ts                # API 契约类型（Post / User / Agent / ApiKey …）
│        ├─ schemas.ts              # 公共 Zod schema（与后端 DTO 同源）
│        ├─ codes.ts                # 错误码枚举、响应规范常量
│        └─ constants.ts            # 角色枚举、状态枚举、分页默认值
│
├─ backend/                         # @agentblog/backend：后端应用（见第二节）
│  └─ …
│
└─ frontend/                        # @agentblog/frontend：前端应用（前端同学填充）
   └─ package.json
```

### 1.1 `@agentblog/shared` 的职责边界

| 该放 | 不该放 |
|------|--------|
| API 响应 payload 的 TS 类型（`Post`、`UserDTO`…） | 后端内部实现（repository 返回的原始行） |
| 前后端共用的 Zod schema（如 `createPostSchema`） | 仅后端用的 DB 层 schema（Drizzle table 定义留在 backend） |
| 错误码枚举、统一响应结构的常量 | 业务逻辑、数据库访问 |
| 角色/状态等枚举（`Role`、`PostStatus`） | Hono 中间件、AI/MCP 工具实现 |

> 📌 **单一真相源**：同一字段（如 `PostStatus = 'draft' | 'published'`）只在 shared 定义一次，后端 Drizzle schema 与前端 UI 都从 `@agentblog/shared` 引入。前端不再手写重复类型。

### 1.2 工作区依赖关系

```
frontend ──depends on──▶ @agentblog/shared ◀──depends on── backend
```

- `backend/package.json`：`"@agentblog/shared": "workspace:*"`
- `frontend/package.json`：`"@agentblog/shared": "workspace:*"`
- `shared` 不依赖任何其它 workspace（纯契约包，零依赖或仅 `zod`）。

> 💡 `workspace:*` 是 Bun/npm/pnpm 通用的 workspace 协议，发布时会被替换为真实版本号；当前 private 包不会发布，仅本地链接。

> 💡 **为什么 `routes/` 和 `modules/` 分开？**
> `routes` 是 HTTP 协议层（路径、方法、状态码、JSON）；`modules` 是纯业务逻辑（可被 HTTP、MCP、AI 流式三种入口复用）。这样 MCP 工具能直接调 `postService.create()`，不必走 HTTP 自调用。

## 三、分层职责矩阵

| 层 | 文件后缀 | 允许做 | 禁止做 |
|----|----------|--------|--------|
| **共享契约** | `packages/shared/src/*` | 定义 API 类型、公共 Zod schema、错误码、枚举 | 业务逻辑、DB 访问、框架耦合 |
| **路由层** | `*.routes.ts` | 定义路由、挂中间件、Zod 校验入参、调 service、返回 `ok(c, data)` | 直接操作数据库；写业务规则 |
| **服务层** | `*.service.ts` | 编排业务、权限校验、事务、计费、调 repository | 读 `Hono` 的 Context；返回 HTTP 响应 |
| **数据层** | `*.repository.ts` | 纯 Drizzle 查询、insert/update/delete | 写业务规则；调其他 service |
| **Schema** | `*.schema.ts` | 定义 Zod schema 与导出类型 | 含运行时逻辑 |
| **中间件** | `*.ts (middlewares/)` | 解析/校验/拦截，向 `c.var` 注入上下文 | 写业务流程 |
| **库** | `*.ts (lib/)` | 纯函数工具（slug/hash/jwt） | 访问数据库 |

> 📌 **铁律**：依赖方向只能 `routes → services → repositories → db`，不得反向。`@agentblog/shared` 可被任意层引用（它是契约，不是依赖环）。service 之间可互相调用（小心循环依赖），repository 之间不得互相调用。

## 四、一个模块的标准结构（以 post 为例）

```
modules/post/
├─ post.schema.ts        # Zod：CreatePostDTO / UpdatePostDTO / PostQueryDTO
├─ post.repository.ts    # findById / findBySlug / list / insert / update / delete
├─ post.service.ts       # createPost / getPost / listPosts / updatePost / deletePost
└─ post.types.ts         # Post 实体类型（从 Drizzle 推导）
```

### 4.1 schema 即类型来源（`post.schema.ts`）

```ts
import { z } from 'zod'

export const createPostSchema = z.object({
  title: z.string().min(1, '标题不能为空'),
  summary: z.string().optional(),
  content: z.string().min(1),
  coverUrl: z.string().url().optional(),
  status: z.enum(['draft', 'published']).default('draft'),
  tagIds: z.array(z.number().int().positive()).default([]),
})
export type CreatePostDTO = z.infer<typeof createPostSchema>

export const updatePostSchema = createPostSchema.partial()
export type UpdatePostDTO = z.infer<typeof updatePostSchema>
```

> 💡 DTO 类型永远从 Zod schema 推导，不要手写 interface——保持单一真相源。
>
> 💡 **与 `@agentblog/shared` 的分工**：跨前后端复用的契约（如 `PostStatus` 枚举、API 响应 payload 类型、错误码）放共享包；仅在 backend 内部使用的 DTO（如 `CreatePostDTO`，含 `tagIds` 这类前端表单也会用但 service 内部也消费的）看情况——若前端表单直接复用则上提到 shared，否则留模块内。原则：**同一份 schema 只定义一次**。

### 4.2 repository（纯查询）

```ts
// modules/post/post.repository.ts
import { eq } from 'drizzle-orm'
import { db } from '@/db/client'
import { posts, postTags } from '@/db/schema'

export const postRepository = {
  async findById(id: number) {
    const rows = await db.select().from(posts).where(eq(posts.id, id)).limit(1)
    return rows[0] ?? null
  },
  async findBySlug(slug: string) {
    const rows = await db.select().from(posts).where(eq(posts.slug, slug)).limit(1)
    return rows[0] ?? null
  },
  // list / insert / update / delete ...
}
```

### 4.3 service（业务编排）

```ts
// modules/post/post.service.ts
import { postRepository } from './post.repository'
import { tagRepository } from '@/modules/tag/tag.repository'
import { generateSlug } from '@/lib/slug'
import { HttpError } from '@/lib/errors'

export const postService = {
  async create(authorId: number, authorType: 'user' | 'agent', dto: CreatePostDTO) {
    const slug = dto.status === 'published' ? generateSlug(dto.title) : null
    return db.transaction(async (tx) => {
      const [post] = await tx.insert(posts).values({ ...dto, slug, authorId, authorType }).returning()
      if (dto.tagIds.length) {
        await tx.insert(postTags).values(dto.tagIds.map((tagId) => ({ postId: post.id, tagId })))
      }
      return post
    })
  },

  async getBySlug(slug: string) {
    const post = await postRepository.findBySlug(slug)
    if (!post) throw HttpError.notFound('文章不存在')
    return post
  },
  // ...
}
```

### 4.4 routes（HTTP 边界）

```ts
// routes/posts.routes.ts
import { Hono } from 'hono'
import { zValidator } from '@hono/zod-validator'
import { authMiddleware } from '@/middlewares/auth'
import { requireRole } from '@/middlewares/rbac'
import { postService } from '@/modules/post/post.service'
import { createPostSchema } from '@/modules/post/post.schema'
import { ok } from '@/lib/response'

export const posts = new Hono()
  .use('*', authMiddleware)           // 所有文章接口都要登录
  .post('/', requireRole('user', 'admin'), zValidator('json', createPostSchema), async (c) => {
    const user = c.var.user
    const dto = c.req.valid('json')
    const post = await postService.create(user.id, 'user', dto)
    return ok(c, post, 201)
  })
  .get('/:slug', async (c) => {
    const post = await postService.getBySlug(c.req.param('slug'))
    return ok(c, post)
  })
```

> 💡 Hono 链式 API：`.use()` 挂中间件后可继续 `.get/.post`，类型安全会推导出路径参数。

## 六、应用组装（`backend/src/app.ts`）

```ts
import { Hono } from 'hono'
import { logger } from 'hono/logger'
import { api } from '@/routes'
import { mcpHandler } from '@/mcp/handler'
import { errorHandler } from '@/middlewares/error-handler'
import { requestId } from '@/middlewares/request-id'

export const app = new Hono()

// 全局中间件（顺序敏感）
app.use('*', requestId())
app.use('*', logger())
app.use('*', errorHandler())   // 见 12，必须放最外层

// 健康检查
app.get('/health', (c) => c.json({ ok: true, data: { status: 'running' } }))

// 业务路由
app.route('/api', api)

// MCP Server（独立入口，不走 /api，自带鉴权）
app.route('/mcp', mcpHandler)
```

> 💡 `errorHandler` 用 `app.onError` 实现更佳（见 [12](./12-统一错误处理与响应规范.md)），上例为示意。

## 七、命名约定

| 类型 | 约定 | 示例 |
|------|------|------|
| 文件 | kebab-case | `api-key.service.ts` |
| 变量/函数 | camelCase | `createPost`、`postRepository` |
| 类型/接口 | PascalCase | `CreatePostDTO`、`Post` |
| Zod schema | camelCase + `Schema` 后缀 | `createPostSchema` |
| 枚举值 | snake_case（与 DB 对齐） | `'draft'`、`'published'`、`'super_admin'` |
| 路由变量 | kebab-case 路径 | `/api/api-keys/:id` |
| Hono context 注入 | `c.var.<name>` | `c.var.user`、`c.var.apiKeyUser` |

## 八、错误传播约定

- service / repository 抛 `HttpError`（见 [12](./12-统一错误处理与响应规范.md)）。
- 路由层不 try/catch 业务错误，统一交给 `app.onError`。
- 路由层只在需要把错误**转换成流式**（如 chat）时局部处理。

```ts
// service 抛
throw HttpError.forbidden('无权修改他人文章')
// 自动被 errorHandler 转成
// { "ok": false, "error": { "code": "FORBIDDEN", "message": "无权修改他人文章" } }
```
