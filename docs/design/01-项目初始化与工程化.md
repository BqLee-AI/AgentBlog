## 一、前置要求

```bash
# 检查 Bun 是否安装（需求锁定运行时）
bun --version   # 期望 ≥ 1.1.0
node --version  # 仅 Lint/编辑器需要，Bun 运行时不依赖 node
```

## 二、Monorepo 方案选型

**采用 Bun Workspaces**——与运行时一致、零额外工具、天然契合需求 §开发分工「统一 Bun 包管理」约定。

| 候选 | 是否选用 | 理由 |
|------|----------|------|
| **Bun Workspaces** | ✅ 选定 | 原生支持、零配置开销、依赖可共享；3 天紧凑项目无任务编排痛点 |
| Turborepo | ❌ | 任务编排与缓存收益对「前后两端 + 一个共享包」过重，引入成本 > 收益 |
| 纯目录分区 | ❌ | 各包独立 `package.json` 不联动，无法共享依赖与类型，契约易漂移 |

**Workspace 划分（共三个工作区）：**

| 工作区 | 包名 | 角色 |
|--------|------|------|
| `packages/shared` | `@agentblog/shared` | 前后端共享：API 契约类型、Zod schema、错误码、常量 |
| `backend` | `@agentblog/backend` | 后端应用（本文焦点） |
| `frontend` | `@agentblog/frontend` | 前端应用（本次先占位，前端同学填充） |

> 📌 **shared 包的价值**：后端 DTO、MCP 工具入参、前端表单校验三者共用同一份 Zod schema（`@agentblog/shared`），避免契约漂移——这正是需求 §1.2「机器可读 / 引用稳定」在工程上的落点。

## 三、初始化 Monorepo 骨架

> ⚠️ 本篇操作默认你站在**仓库根** `D:\aaa\AgentBlog` 下，而非 `backend/` 内。后端专属命令会显式 `cd backend`。

### 3.1 根目录

```bash
cd D:\aaa\AgentBlog
```

```jsonc
// package.json（仓库根，手动创建）
{
  "name": "agentblog",
  "private": true,
  "workspaces": ["packages/*", "backend", "frontend"],
  "scripts": {
    "dev": "bun run --filter '*' dev",
    "dev:backend": "bun run --filter @agentblog/backend dev",
    "dev:frontend": "bun run --filter @agentblog/frontend dev",
    "build": "bun run --filter '*' build",
    "typecheck": "bun run --filter '*' typecheck",
    "lint": "bun run --filter '*' lint"
  }
}
```

### 3.2 共享包 `packages/shared`

```bash
mkdir -p packages/shared/src
```

```jsonc
// packages/shared/package.json
{
  "name": "@agentblog/shared",
  "version": "0.0.0",
  "private": true,
  "type": "module",
  "exports": {
    ".": "./src/index.ts"
  },
  "scripts": {
    "typecheck": "tsc --noEmit"
  }
}
```

```ts
// packages/shared/src/index.ts（初始占位，后续按模块填充）
export * from './types'
export * from './schemas'
export * from './codes'
```

> 💡 shared 包暂只导出 TS 源码（`exports` 指向 `./src/index.ts`），不编译。Bun 与前端构建链都能直接消费 TS 源，省去 build 步骤；若后续构建链不支持，再加 `tsup` 编译。

### 3.3 后端工作区 `backend`

```bash
mkdir backend && cd backend
bun init -y   # 生成 backend/package.json、tsconfig.json、index.ts，随后覆盖
cd ..
```

## 四、安装后端依赖

> 在 `backend/` 下执行。Bun workspace 会自动 hoist 到根 `node_modules`。

```bash
cd backend

# 引用共享包（workspace 协议）
bun add @agentblog/shared@workspace:*

# 核心运行时
bun add hono @hono/zod-validator zod
bun add drizzle-orm
bun add @ai-sdk/openai ai          # Vercel AI SDK（按实际 provider 调整）
bun add @modelcontextprotocol/sdk  # MCP Server
bun add bcryptjs                   # 密码哈希

# 开发依赖
bun add -d drizzle-kit              # 迁移工具
bun add -d @types/bcryptjs
bun add -d typescript
bun add -d eslint prettier
bun add -d @typescript-eslint/eslint-plugin @typescript-eslint/parser
bun add -d tsx                      # 脚本执行（可选，Bun 本身可跑 ts）
bun add -d dotenv-cli               # 注入环境变量跑迁移/脚本
```

> 💡 SQLite 驱动：Drizzle 在 Bun 下推荐用 `bun:sqlite`（Bun 内置，零安装），不需要 `better-sqlite3`。

## 五、TypeScript 配置（根 base + 各包继承）

monorepo 下 TS 配置分两层：**根 `tsconfig.base.json` 放公共基线**，各 workspace 用 `extends` 继承并加自身专属项。避免在每个包里重复 60 行相同配置。

### 5.1 根基线 `tsconfig.base.json`

```jsonc
// tsconfig.base.json（仓库根）
{
  "compilerOptions": {
    "target": "ES2022",
    "module": "ESNext",
    "moduleResolution": "Bundler",   // Bun/Hono 生态推荐
    "lib": ["ES2022"],

    "strict": true,                  // TS 最佳实践基线
    "noUncheckedIndexedAccess": true,
    "noImplicitOverride": true,
    "noFallthroughCasesInSwitch": true,
    "exactOptionalPropertyTypes": true,

    "esModuleInterop": true,
    "allowSyntheticDefaultImports": true,
    "resolveJsonModule": true,
    "isolatedModules": true,
    "verbatimModuleSyntax": true,    // 强制 type-only import

    "skipLibCheck": true,
    "noEmit": true                   // Bun 直接跑源码，不编译产物
  }
}
```

### 5.2 后端 `backend/tsconfig.json`

```jsonc
{
  "extends": "../tsconfig.base.json",
  "compilerOptions": {
    "types": ["bun-types"],
    "baseUrl": ".",
    "paths": {
      "@/*": ["src/*"],
      "@agentblog/shared": ["../packages/shared/src/index.ts"],
      "@agentblog/shared/*": ["../packages/shared/src/*"]
    }
  },
  "include": ["src/**/*.ts", "drizzle.config.ts"],
  "exclude": ["node_modules", "dist"]
}
```

### 5.3 共享包 `packages/shared/tsconfig.json`

```jsonc
{
  "extends": "../../tsconfig.base.json",
  "compilerOptions": {
    "baseUrl": ".",
    "paths": {}
  },
  "include": ["src/**/*.ts"]
}
```

**为什么这样配**：
- `extends` 继承根基线：所有 workspace 的 strict / Bundler / verbatimModuleSyntax 保持一致，单一真相源。
- `Bundler` 模式解析：Hono、Vercel AI SDK、MCP SDK 都用 ESM + exports map。
- `@/*` 本地别名 + `@agentblog/shared` 包别名：避免深层相对路径 `../../../`，共享包导入写包名而非相对路径。
- shared 包通过 `paths` 映射到源码，编辑器跳转直达源文件，无需先 build。

```js
  { files: ['src/**/*.ts'] },
  {
    languageOptions: {
      parser: tsparser,
      parserOptions: { project: './tsconfig.json' },

  },
  {

      '@typescript-eslint/no-unused-vars': ['warn', { argsIgnorePattern: '^_' }],
      '@typescript-eslint/no-floating-promises': 'error',  // 防止漏 await

    },
  },
  { ignores: ['dist/**', 'node_modules/**'] },
]
```

```jsonc
{
```

### 7.3 类型安全的配置读取（`src/config/env.ts`）

> 📌 这是 TS 最佳实践的关键：**用 Zod 在启动时校验环境变量，启动即失败**，而不是运行时才报 `undefined`。

```ts
import { z } from 'zod'

const envSchema = z.object({
  NODE_ENV: z.enum(['development', 'production', 'test']).default('development'),
  PORT: z.coerce.number().default(3000),
  DATABASE_URL: z.string().default('./data/app.db'),
  JWT_SECRET: z.string().min(32, 'JWT_SECRET 至少 32 字符'),
  JWT_EXPIRES_IN: z.string().default('7d'),
  SUPER_ADMIN_USERNAME: z.string(),
  SUPER_ADMIN_PASSWORD: z.string(),
  CREDITS_PER_MCP_CALL: z.coerce.number().positive().default(1),
  TOKENS_PER_CREDIT: z.coerce.number().positive().default(1000),
  AI_PROVIDER: z.enum(['openai', 'anthropic']).default('openai'),
  OPENAI_API_KEY: z.string().optional(),
  AI_MODEL: z.string().default('gpt-4o-mini'),
  UPLOAD_DIR: z.string().default('./data/uploads'),
  UPLOAD_MAX_SIZE_MB: z.coerce.number().positive().default(5),
})

const parsed = envSchema.safeParse(process.env)
if (!parsed.success) {
  console.error('❌ 环境变量校验失败：')
  console.error(parsed.error.flatten().fieldErrors)
  process.exit(1)
}

export const env = parsed.data
export type Env = typeof env
```

使用方式：所有地方都从 `@/config/env` 读 `env`，禁止直接用 `process.env.XXX`。

## 八、Git 分支策略（需求 §开发分工指定）

```
feature/* → develop → main
```

| 分支 | 用途 | 谁合并 |
|------|------|--------|
| `main` | 生产可发布 | 仅 PM/lead 合并 |
| `develop` | 集成分支 | 后端/前端各自合入 |
| `feature/post-crud` | 功能开发 | 完成后 PR 到 develop |

> 💡 命名约定：`feature/<模块>-<动作>`，如 `feature/rbac-middleware`、`feature/mcp-server`。

```bash
git init
git checkout -b develop
# 功能分支示例
git checkout -b feature/backend-scaffold
```

## 九、入口文件骨架（`backend/src/index.ts`）

完成本篇后，先放一个最小可跑的入口验证环境通不通（完整 app 组装见 [02](./02-目录结构与分层规范.md)）：

```ts
import { Hono } from 'hono'
import { env } from '@/config/env'

const app = new Hono()

app.get('/health', (c) => c.json({ ok: true, data: { status: 'running' } }))

export default {
  port: env.PORT,
  fetch: app.fetch,
}
```

```bash
bun run dev
# 访问 http://localhost:3000/health → { "ok": true, "data": { "status": "running" } }
```

> 💡 Hono 的 Bun 适配：直接 `export default { port, fetch }`，无需 `Bun.serve()` 包装。
