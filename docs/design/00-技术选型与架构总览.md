# 00 · 技术选型与架构总览

> 本文回答三个问题：**为什么这样选、整体怎么分、请求怎么流。**
> 所有后续文档都建立在本篇的架构决策之上。

---

## 一、技术栈总览

| 层 | 选型 | 版本基线 | 选型理由 |
|----|------|----------|----------|
| 运行时 | **Bun** | ≥ 1.1 | 原生 TS、启动快、内置 SQLite 驱动、内置测试、内置包管理 |
| Web 框架 | **Hono** | ≥ 4.x | 轻量、Web 标准 Request/Response、类型安全路由、流式友好、Bun 适配 |
| 数据库 | **SQLite** | (Bun 内置) | 单文件、零运维、千级数据量足够（需求 §5） |
| ORM | **Drizzle ORM** | ≥ 0.30 | TS 优先、SQL-like、迁移清晰、类型推导强、与 Bun/Hono 契合 |
| 校验 | **Zod** | ≥ 3.x | schema 即类型、与 Hono validator 无缝集成 |
| AI | **Vercel AI SDK** | ≥ 3.x（`ai` + `@ai-sdk/*`） | 需求指定；提供 `streamText` 与 UI Message 协议 |
| MCP | **`@modelcontextprotocol/sdk`** | ≥ 1.x | 官方 SDK，作为 MCP Server 暴露文章 CRUD 工具 |
| 鉴权 | JWT（`hono/utils/jwt`）+ API Key | — | 用户会话走 JWT，外部 MCP 走 Header API Key |
| 密码 | **bcrypt**（`bcryptjs` 纯 JS 或 Bun 原生） | — | 加盐哈希，不可逆 |
| 测试 | **Bun test**（内置 `bun:test`） | — | 零配置、与运行时一致 |
| 包管理 | **Bun**（`bun install`） | — | 需求 §开发分工指定统一 Bun 包管理 |
| 仓库结构 | **Bun Workspaces（monorepo）** | — | 前后端 + `@agentblog/shared` 共享契约包，依赖 hoist |

> 📌 **强约束（来自需求文档）**：Bun + Hono + SQLite + Vercel AI SDK + MCP。这五项不可替换。
> 💡 ORM/校验/测试/仓库结构 未在需求中锁定，本套文档选 Drizzle + Zod + Bun test + Bun Workspaces 为 TS 最佳实践。monorepo 方案对比见 [01 §二](./01-项目初始化与工程化.md)。

## 二、四大子系统拆分

需求 §2.4「系统拆分」明确划分四部分。后端代码按这四部分组织成独立的入口与模块：

```
┌─────────────────────────────────────────────────────────────┐
│                     AgentBlog 后端                           │
│                                                              │
│  ┌──────────────────┐   ┌──────────────────────────────┐   │
│  │ ① 写作端 Web API │   │ ② 内容服务端（核心业务）       │   │
│  │   /api/*         │   │   用户/文章/标签/Agent/Credits │   │
│  │   给人：后台管理  │   │   给 Agent：被 ③④ 复用         │   │
│  └──────────────────┘   └──────────────────────────────┘   │
│           │                          ▲  ▲                    │
│           │                          │  │                    │
│  ┌────────┴────────┐   ┌─────────────┘  └─────────────┐    │
│  │ ③ Agent 运行环境 │   │ ④ MCP Server                  │    │
│  │   /api/chat 流式  │   │   /mcp 文章 CRUD 工具          │    │
│  │   Vercel AI SDK  │   │   Header API Key 鉴权          │    │
│  │   Token 计费      │   │   按次计费                     │    │
│  └─────────────────┘   └───────────────────────────────┘    │
└─────────────────────────────────────────────────────────────┘
```

| 子系统 | HTTP 入口 | 鉴权 | 计费 | 谁来实现 |
|--------|-----------|------|------|----------|
| ① 写作端 Web API | `/api/*` | JWT（Cookie/Authorization） | 不计费 | 江涵 + 李钊 |
| ② 内容服务端 | 被 ①③④ 复用 | —（内部服务层） | — | 江涵 + 李钊 |
| ③ Agent 运行环境 | `/api/chat` | JWT（在线，用户态） | 按 Token | 黎殊涵 |
| ④ MCP Server | `/mcp` | Header API Key | 按次 | 江涵 |

> 💡 ①③④ 三个 HTTP 入口共用同一个 Bun 进程（Hono app），通过路由前缀区分；② 是它们共用的 service 层，不独立暴露 HTTP。

## 三、分层架构（Hono 函数式分层）

Hono 不是 NestJS 那种装饰器/DI 框架。TS 最佳实践走的是**函数式四层 + 按业务域组织**：

```
HTTP 请求
   │
   ▼
┌──────────────────────────────────────────────────────┐
│ routes / *.routes.ts   路由层                         │
│   - 定义路径、HTTP 方法、挂中间件                      │
│   - 用 Zod schema 校验 body/query/param              │
│   - 解析当前用户（auth 中间件）                        │
│   - 调 service，把结果包装成统一响应                   │
└──────────────────────────────────────────────────────┘
   │  调用
   ▼
┌──────────────────────────────────────────────────────┐
│ services / *.service.ts  业务层                       │
│   - 编排业务规则（权限校验、计费、作者归属）            │
│   - 事务边界                                          │
│   - 调一个或多个 repository                           │
└──────────────────────────────────────────────────────┘
   │  调用
   ▼
┌──────────────────────────────────────────────────────┐
│ repositories / *.repository.ts  数据层                │
│   - 只管 CRUD，纯 Drizzle 查询                        │
│   - 不含业务规则                                      │
│   - 返回实体或实体数组                                │
└──────────────────────────────────────────────────────┘
   │  使用
   ▼
┌──────────────────────────────────────────────────────┐
│ db /  Drizzle client + schema                         │
└──────────────────────────────────────────────────────┘
```

**横向贯穿的「横切关注点」放在 `middlewares/`**：auth、rbac、error-handler、request-id、logger、rate-limit。它们通过 `app.use('*')` 或路由级 `app.use('/api/posts/*', authMiddleware)` 挂载。

**跨前后端的「API 契约」放在 monorepo 共享包 `@agentblog/shared`**：类型、Zod schema、错误码、枚举只在此定义一次，后端 DTO 与前端表单共用，保证契约不漂移（需求 §1.2「机器可读 / 引用稳定」的工程落点）。

详见 [02-目录结构与分层规范](./02-目录结构与分层规范.md)。

## 四、请求链路（三张关键时序图）

### 4.1 普通用户发布文章

```
前端              routes/posts        services/post        repositories/post
  │  POST /api/posts  │                    │                       │
  │ ──────────────────▶│                    │                       │
  │                    │ auth 中间件解析 user │                       │
  │                    │ Zod 校验 body       │                       │
  │                    │ createPost(req)    │                       │
  │                    │ ───────────────────▶│                       │
  │                    │                    │ 校验作者归属           │
  │                    │                    │ 生成 slug（若发布）    │
  │                    │                    │ db.transaction(…)     │
  │                    │                    │ insertPost() ─────────▶│
  │                    │                    │ syncTags()  ──────────▶│ (post_tag)
  │                    │                    │ ◀─────────────────────│
  │                    │ ◀──────────────────│ 返回 post             │
  │ { ok,data: post }  │                    │                       │
  │ ◀──────────────────│                    │                       │
```

### 4.2 在线 Agent 对话（流式 + Token 计费）

```
前端 useChat       routes/chat          services/agent-run     AI SDK
  │ POST /api/chat   │                       │                    │
  │ (流式)           │                       │                    │
  │ ────────────────▶│ auth 解析 user         │                    │
  │                  │ 预检 credits>0        │                    │
  │                  │ streamText(req) ──────▶│ 注入 System Prompt  │
  │                  │                       │ + 内置工具          │
  │                  │                       │ streamText() ──────────▶ LLM
  │                  │                       │ ◀──────────────── 流式 token
  │ ◀════════════════│ ◀═════════════════════│ 逐块透传 UI Message │
  │  (SSE/Data)      │ 流结束后              │                    │
  │                  │ onResult 回调 ────────▶│ 读 totalTokens      │
  │                  │                       │ 原子扣 credits      │
  │                  │                       │ 写 credit_log       │
  │                  │ 不存聊天记录           │                    │
```

### 4.3 外部 MCP 调用（Header API Key + 按次计费）

```
外部 Agent          MCP Server(本系统)        services
(MCP 客户端)        /mcp                       │
  │                  │                          │
  │ tools/call       │ 解析 Header X-API-Key    │
  │  create_post     │ apiKey 鉴权中间件 ───────▶ validateApiKey()
  │ ────────────────▶│ 401 if 无效              │ → 返回 user(=Agent 主人)
  │                  │ 计费中间件：预检 credits │
  │                  │ dispatch 到对应工具 ─────▶ postService.create()
  │                  │ 成功后：按次计 1 credit  │ chargeCredits(1)
  │                  │ 写 credit_log            │
  │ ◀────────────────│ MCP 响应                 │
```

## 五、与前端、外部的关系边界

| 调用方 | 接入点 | 协议 | 鉴权 |
|--------|--------|------|------|
| 写作后台（人） | `/api/*` | REST + JSON | JWT |
| 阅读页（人） | `/api/posts`、`/api/posts/:slug` | REST + JSON | 无（公开已发布） |
| 在线对话（人通过平台） | `/api/chat` | SSE 流式（UI Message） | JWT |
| 外部 MCP 客户端 | `/mcp` | JSON-RPC over HTTP/SSE | Header `X-API-Key` |

> 📌 **引用稳定**：文章 slug 一经发布永久不变（需求 §1.2/§5）。MCP 的 `get_post` 支持按 slug 读取，保证 Agent 长期可引用。

## 六、非功能性约束（来自需求 §5）

后端设计必须满足：

| 约束 | 后端落地方式 |
|------|--------------|
| 机器可读 | MCP 返回 JSON；REST 统一 `{ ok, data, error }`（见 [12](./12-统一错误处理与响应规范.md)） |
| 性能：P95 ≤ 1s | 关键查询加索引（slug/username/author_id）；列表分页必加 limit |
| 引用稳定 | slug 不可变，发布后 update 接口禁止改 slug（见 [06](./06-文章模块.md)） |
| 计费正确 | credits 扣减用数据库事务 + 行锁，防超扣（见 [08](./08-Credits-计费与原子扣减.md)） |
| 隐私：不存记录 | 在线对话流不落库；不写 message 表（见 [10](./10-在线-Agent-运行环境.md)） |
| 易用性 | 所有错误提示中文（见 [12](./12-统一错误处理与响应规范.md)） |
