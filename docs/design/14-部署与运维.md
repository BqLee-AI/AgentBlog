# 14 · 部署与运维

> 本篇定义生产部署、进程守护、日志、备份、监控。
> 对应任务 18 的「部署」部分。
>
> **技术栈约束**：Bun 运行时 + SQLite 单文件数据库。

---

## 一、部署形态选择

| 形态 | 适用 | 说明 |
|------|------|------|
| **单机 Bun + SQLite** | v1 默认 | 一台服务器，`bun run start`，SQLite 文件本地 |
| Docker Compose | 推荐生产 | 容器化 + 卷挂载 SQLite，易迁移 |
| 反向代理 + Bun | 高并发 | Nginx 在前，处理 TLS/静态，Bun 在后 |

> 💡 SQLite 单机足以支撑需求 §5 的「千级数据量」。如未来扩多机，再换 PostgreSQL（Drizzle 迁移成本低）。

## 三、进程守护

Bun 自身不是 daemon，生产需用进程管理器保证崩溃自重启。

### 3.1 方案 A：systemd（Linux 推荐）

```ini
# /etc/systemd/system/agentblog.service
[Unit]
Description=AgentBlog Backend
After=network.target

[Service]
Type=simple
User=agentblog
WorkingDirectory=/opt/agentblog
EnvironmentFile=/opt/agentblog/.env.production
ExecStart=/home/agentblog/.bun/bin/bun run src/index.ts
Restart=always
RestartSec=3

[Install]
WantedBy=multi-user.target
```

```bash
sudo systemctl enable agentblog
sudo systemctl start agentblog
journalctl -u agentblog -f   # 看日志
```

### 3.2 方案 B：PM2（跨平台）

```bash
pm2 start "bun run src/index.ts" --name agentblog
pm2 save
pm2 startup
```

> 💡 3 天项目优先 PM2（简单）。长期运行用 systemd 更稳。

## 四、Docker 化

### 4.1 Dockerfile

```dockerfile
FROM oven/bun:1.1 AS base
WORKDIR /app

# 依赖层（利用缓存）
COPY package.json bun.lockb ./
RUN bun install --production --frozen-lockfile

# 代码层
COPY src ./src
COPY drizzle ./drizzle
COPY drizzle.config.ts tsconfig.json ./

# 数据卷
VOLUME ["/app/data"]
ENV DATABASE_URL=/app/data/app.db
ENV UPLOAD_DIR=/app/data/uploads

EXPOSE 3000

# 启动：先迁移再启动
CMD ["sh", "-c", "bun run db:migrate && bun run src/index.ts"]
```

> 💡 多阶段构建可进一步减小镜像（builder 跑 build，runtime 只装产物）。Bun 直接跑源码，单阶段也够用。

### 4.2 docker-compose.yml

```yaml
version: '3.9'
services:
  backend:
    build: ./backend
    ports:
      - "3000:3000"
    volumes:
      - ./data:/app/data          # SQLite 与上传文件持久化
      - ./.env.production:/app/.env:ro
    restart: unless-stopped
    healthcheck:
      test: ["CMD", "curl", "-f", "http://localhost:3000/health"]
      interval: 30s
      timeout: 5s
      retries: 3

  # 可选：前端
  # frontend:
  #   build: ./frontend
  #   ports: ["80:80"]
```

```bash
docker compose up -d
docker compose logs -f backend
```

## 五、反向代理（Nginx）

```nginx
server {
    listen 443 ssl http2;
    server_name api.your-blog.com;

    ssl_certificate     /etc/ssl/cert.pem;
    ssl_certificate_key /etc/ssl/key.pem;

    # 后端 API + MCP + 流式
    location / {
        proxy_pass http://127.0.0.1:3000;
        proxy_http_version 1.1;
        proxy_set_header Host $host;
        proxy_set_header X-Real-IP $remote_addr;
        proxy_set_header X-Forwarded-For $proxy_add_x_forwarded_for;
        proxy_set_header X-Forwarded-Proto $scheme;

        # SSE/流式关键：禁用缓冲，超时调长
        proxy_buffering off;
        proxy_read_timeout 300s;
        proxy_cache off;
    }

    # 静态上传文件（可选，Nginx 直服）
    location /uploads/ {
        alias /opt/agentblog/data/uploads/;
        expires 7d;
    }
}
```

> 📌 **流式必读**：`proxy_buffering off` 与长超时是 SSE 正常工作的关键，否则 `/api/chat` 会被 Nginx 缓冲导致前端收不到流。

## 六、日志

### 6.1 结构化日志（pino）

```bash
bun add pino
```

```ts
// lib/logger.ts
import pino from 'pino'
import { env } from '@/config/env'

export const logger = pino({
  level: env.NODE_ENV === 'production' ? 'info' : 'debug',
  transport: env.NODE_ENV === 'development'
    ? { target: 'pino-pretty' }
    : undefined,
})
```

```ts
// 关键位置打点
logger.info({ userId, action: 'login' }, '用户登录')
logger.warn({ userId, credits, amount }, '额度不足')
logger.error({ err, requestId }, '请求失败')
```

### 6.2 关键日志点

| 事件 | 级别 | 字段 |
|------|------|------|
| 登录成功/失败 | info/warn | userId, username, ip |
| 文章发布 | info | postId, authorType |
| Credits 扣减 | info | userId, delta, type, balance |
| 余额不足 | warn | userId, action |
| API Key 签发/吊销 | info | agentId, keyId |
| MCP 调用 | info | agentId, tool, userId |
| 在线对话 | info | userId, tokenUsage |
| 未处理错误 | error | err, requestId |

### 6.3 请求日志中间件

```ts
import { logger as honoLogger } from 'hono/logger'
// 或自定义带 pino
app.use('*', async (c, next) => {
  const start = Date.now()
  await next()
  const ms = Date.now() - start
  logger.info({ method: c.req.method, path: c.req.path, status: c.res.status, ms }, '请求')
})
```

## 七、备份

SQLite 单文件，备份极简：

```bash
# 在线备份（用 sqlite 命令，不停服）
sqlite3 ./data/app.db ".backup '/backup/app-$(date +%F).db'"

# 或用 Bun 脚本定时
# crontab: 每天凌晨 3 点
0 3 * * * sqlite3 /opt/agentblog/data/app.db ".backup '/backup/app-$(date +\%F).db'" && find /backup -mtime +7 -delete
```

> 💡 `.backup` 是 SQLite 的在线一致性备份，不停服、可热备。比 `cp` 文件安全（cp 可能抓到写一半的状态）。
> 💡 上传文件目录 `data/uploads/` 也需定期同步到对象存储/异地。

## 八、监控（v1 最小集）

| 指标 | 方式 |
|------|------|
| 进程存活 | systemd/PM2 + healthcheck |
| `/health` 可达 | UptimeRobot / curl 定时 |
| 磁盘空间 | SQLite + 上传文件会涨 |
| 错误率 | 日志聚合（grep `level:50`） |

```ts
// /health 接口增强
app.get('/health', async (c) => {
  // 探测 DB
  try {
    await db.run(sql`SELECT 1`)
    return c.json({ ok: true, data: { status: 'running', db: 'ok' } })
  } catch (e) {
    return c.json({ ok: false, error: { code: 'DB_DOWN', message: '数据库不可用' } }, 503)
  }
})
```

## 九、安全清单（上线前）

- [ ] `JWT_SECRET` 为强随机（`openssl rand -hex 32`）
- [ ] 超管密码非默认（非 `admin12345`）
- [ ] `.env.production` 不进 git
- [ ] CORS 仅允许前端域名
- [ ] HTTPS 强制（Nginx TLS）
- [ ] API Key 明文不落日志（只 log prefix）
- [ ] 错误响应生产环境不泄露堆栈
- [ ] 上传大小/类型限制生效
- [ ] 限流（防 MCP 滥用）

### 限流示例

```ts
import { rateLimit } from './middlewares/rate-limit'
// MCP 入口限流，防 API Key 被盗刷
mcpHandler.use('*', rateLimit({ window: 60, max: 60, keyBy: (c) => c.var.apiKeyId }))
```

> 💡 简单限流可用内存计数；分布式场景上 Redis。v1 内存版够用。

## 十、部署 Runbook（上线步骤）


## 十一、回滚

```bash
# 代码回滚
git checkout <prev-commit>
bun install --production
pm2 restart agentblog

# 数据库回滚（谨慎）
# drizzle-kit 不自动降级，需手动跑 reverse SQL 或恢复备份
sqlite3 ./data/app.db "..."
# 或直接恢复昨日备份
cp /backup/app-2026-07-09.db ./data/app.db
```

> 💡 **credits 相关操作不可轻易回滚**——回滚 DB 会丢当日所有扣费流水，需与业务确认。
